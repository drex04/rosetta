Five interaction bugs to fix across canvas nodes and the LOAD panel.

<SourcePanel key={activeSourceId ?? 'none'} resetSourceSchema={resetSourceSchema} />

### Task 2 — Fix URI input closing the node edit form on click

**Files:** `src/components/nodes/ClassNode.tsx`, `src/components/nodes/SourceNode.tsx`

Both nodes put `onBlur={() => commitHeader()}` on each individual input. Clicking from the label input to the URI input fires blur on the label, commits, and hides the form.

Fix: Remove `onBlur` from individual inputs. Move the commit to the container `div` using a container-level `onBlur` with a `relatedTarget` guard:

```tsx
<div
  className="flex flex-col gap-1 py-1 nodrag w-full"
  onMouseDown={(e) => e.stopPropagation()}
  onBlur={(e) => {
    // Only commit when focus leaves the container entirely
    if (!e.currentTarget.contains(e.relatedTarget as Node)) {
      commitHeader();
    }
  }}
>
  <input
    autoFocus
    // ... remove onBlur
    onKeyDown={...}
    placeholder="Label"
  />
  <input
    // ... remove onBlur
    onKeyDown={...}
    placeholder="URI"
  />
</div>
```

Apply the same change to both `ClassNode.tsx` and `SourceNode.tsx`.

**Done:** User can click between label and URI inputs freely. Form only commits when focus moves outside the form.

### Task 3 — Fix property double-click opening node header instead of property edit

**File:** `src/components/canvas/OntologyCanvas.tsx`

`handleNodeDoubleClick` (line 894–907) unconditionally calls `handleStartEdit(node.id)`, which bumps `editTrigger` → opens the header edit form. This fires even when the double-click originated on a property row.

The property row's inline `onDoubleClick` in `ClassNode.tsx` already correctly sets `editingPropUri`. The canvas-level handler overrides it.

Fix: In `handleNodeDoubleClick`, check whether the double-click event's target is inside a property row before calling `handleStartEdit`:

```tsx
const handleNodeDoubleClick = useCallback(
  (e: React.MouseEvent, node: RFNode) => {
    if (node.type === 'classNode') setActiveRightTab('ONTOLOGY');
    else if (node.type === 'sourceNode') setActiveRightTab('SOURCE');

    // Don't fire header edit if the click originated on a property row
    const target = e.target as HTMLElement;
    if (target.closest('[data-property-row]')) return;

    const onStartEdit = (node.data as { onStartEdit?: (id: string) => void })?.onStartEdit;
    if (typeof onStartEdit === 'function') onStartEdit(node.id);
  },
  [setActiveRightTab],
);
```

Then add `data-property-row` attribute to property row `div`s in `ClassNode.tsx` (line 165) and `SourceNode.tsx` (line 144).

**Done:** Double-clicking a property opens the property name edit. Double-clicking the header opens header edit. No conflict.

### Task 4 — Fix right-click "Rename" on property opening node name edit

**Files:** `src/components/canvas/OntologyCanvas.tsx`, `src/components/nodes/ClassNode.tsx`, `src/components/types/index.ts`

When right-clicking a property row, the event bubbles to the node's `handleContextMenu` which opens the node-level context menu. "Rename" calls `handleStartEdit(nodeId)` → header edit opens.

Fix with a `propRenameTrigger` pattern (mirrors `editTrigger`):

**Step A — ClassNode.tsx:** Add `propRenameTrigger?: { propUri: string; counter: number }` to `ClassData` type. Add a `useEffect` that watches `data.propRenameTrigger` and sets `editingPropUri` + draft state:

```tsx
useEffect(() => {
  if (!data.propRenameTrigger) return;
  const { propUri } = data.propRenameTrigger;
  const prop = data.properties.find((p) => p.uri === propUri);
  if (!prop) return;
  setEditingPropUri(propUri);
  setDraftPropLabel(prop.label ?? prop.uri);
  setDraftPropType(prop.dataType ?? 'xsd:string');
  setPropError('');
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [data.propRenameTrigger]);
```

**Step B — ClassNode.tsx property rows:** Add `onContextMenu` to each property row div that stops propagation and calls `data.onPropContextMenu?.(id, prop.uri, e.clientX, e.clientY)`:

```tsx
onContextMenu={(e) => {
  e.preventDefault();
  e.stopPropagation(); // don't bubble to node context menu
  data.onPropContextMenu?.(id, prop.uri, e.clientX, e.clientY);
}}
```

**Step C — OntologyCanvas.tsx:** Add a `propMenu` state tracking `{ nodeId, propUri, x, y }`. Wire `data.onPropContextMenu` in `augmentedNodes`. Show a small inline dropdown with a single "Rename Property" item that calls `handleStartPropEdit(nodeId, propUri)`. `handleStartPropEdit` bumps `propRenameTrigger` on the node data (same pattern as `handleStartEdit`/`editTrigger`).

Apply equivalent changes to `SourceNode.tsx` for completeness (SourceNode properties are read-only but the pattern should be symmetric for future use).

**Done:** Right-clicking a property shows a property context menu; "Rename Property" makes that property's name inline-editable.
