# Plan 13-04: Visual & Display Fixes

## Context

Five visual/display bugs: MAP tab datatype URIs, accordion overflow, text sizing, Add Source color, and the flashing status bar.

## Tasks

### Task 1 — MAP tab: show short datatype names (xsd:string not full URI)

**File:** `src/components/panels/MappingPanel.tsx`

`getPropRange()` returns `prop.range` raw. For ontology nodes, `prop.range` is stored as the full XSD URI (e.g. `http://www.w3.org/2001/XMLSchema#string`). The mapping list renders it directly.

Fix: Import `shortenRange` from `@/lib/rdf` and wrap both `sourceRange` and `targetRange` display values:

```tsx
// Add to imports:
import { localName, shortenRange } from '@/lib/rdf';

// In the mapping list rows (around line 583 and 593):
{sourceRange ? (
  <span className="font-mono text-muted-foreground text-sm shrink-0">
    {shortenRange(sourceRange)}
  </span>
) : null}
// ...
{targetRange ? (
  <span className="font-mono text-muted-foreground text-sm shrink-0">
    {shortenRange(targetRange)}
  </span>
) : null}
```

`shortenRange` already handles the `STANDARD_NAMESPACES` lookup in `rdf.ts:99-106` — no new logic needed.

**Done:** Mapping list shows `xsd:string`, `xsd:float`, etc. instead of full URIs.

### Task 2 — Fix RML/YARRRML accordion horizontal overflow

**File:** `src/components/panels/OutputPanel.tsx`

The `ScrollArea` wrapper at line 93 (`<ScrollArea className="shrink-0 max-h-[45%]">`) allows vertical scrolling but the accordion content can expand the container width if the pre content is wide.

The `pre` elements already have `whitespace-pre-wrap` which should wrap — the issue is the `Accordion` and its parent `div` inside the ScrollArea don't constrain width.

Fix: Add `overflow-x-hidden` and `min-w-0` to the container div inside the ScrollArea, and ensure the `Accordion` itself doesn't overflow:

```tsx
// Line 94 — add overflow-x-hidden to the inner container:
<div className="flex flex-col gap-3 p-3 overflow-x-hidden min-w-0">
```

Also add `overflow-hidden` to each `AccordionContent` wrapper and ensure the `ScrollArea` inside AccordionContent constrains width:

```tsx
// Both AccordionContent blocks (lines 137, 165):
<AccordionContent className="p-0 overflow-hidden">
  <ScrollArea className="h-[280px]">
    <pre className="px-3 py-2 font-mono text-sm text-foreground bg-muted/30 border-t border-border whitespace-pre-wrap break-all">
```

The `break-all` addition ensures long lines in the Turtle/YAML content wrap rather than extending horizontally.

**Done:** Expanding RML or YARRRML accordion grows downward only, no horizontal scrollbar or overflow.

### Task 3 — Match RML/YARRRML text size to other code panes

**File:** `src/components/panels/OutputPanel.tsx`

The `pre` elements at lines 139 and 167 already use `text-sm`. Verify the CodeMirror-based panes also use `text-sm` (they do via the `lightTheme` in `src/lib/codemirror-theme.ts`). The issue may be that the `pre` element's default browser font size differs from CodeMirror's.

Fix: Ensure both `pre` elements explicitly set `text-sm` (already present) AND the same font family:

```tsx
<pre className="px-3 py-2 font-mono text-sm leading-relaxed text-foreground bg-muted/30 border-t border-border whitespace-pre-wrap break-all">
```

If the mismatch is still visible after this, check `src/lib/codemirror-theme.ts` for any `fontSize` override and match it in the `pre` style. Add `style={{ fontSize: '0.875rem' }}` (14px) as a fallback if Tailwind `text-sm` is being overridden by browser defaults on `pre` elements.

**Done:** RML and YARRRML preview text visually matches the size of text in the Turtle, JSON, and SPARQL code editors.

### Task 5 — Status bar: always visible, no flash

**File:** `src/components/layout/StatusBar.tsx`

Current behavior:
- `idle` → `text-transparent` (bar exists but content invisible)
- `saving` → waits 400ms then shows amber
- `saved` → shows green briefly then back to invisible idle

The 400ms delay + idle transparency causes the content to flash in/out on every save cycle.

Fix: Remove the delayed-show logic and make the bar always show something. Simplify to a direct render:

```tsx
export function StatusBar({ saveStatus }: StatusBarProps) {
  let barClass =
    'h-6 flex items-center justify-center gap-1.5 px-3 shrink-0 text-sm font-medium transition-colors border-t';

  if (saveStatus === 'saving') {
    barClass += ' bg-amber-400/20 border-amber-400/30 text-amber-700 dark:text-amber-400';
  } else if (saveStatus === 'saved' || saveStatus === 'idle') {
    barClass += ' bg-green-500/10 border-green-500/20 text-green-700/70 dark:text-green-400/70';
  } else if (saveStatus === 'error') {
    barClass += ' bg-destructive/15 border-destructive/25 text-destructive';
  } else {
    barClass += ' bg-muted/30 border-border text-muted-foreground';
  }

  return (
    <div className={barClass} aria-live="polite">
      {saveStatus === 'saving' && (
        <>
          <CircleNotchIcon size={12} className="animate-spin" />
          Saving…
        </>
      )}
      {(saveStatus === 'saved' || saveStatus === 'idle') && (
        <>
          <CheckCircleIcon size={12} />
          Saved
        </>
      )}
      {saveStatus === 'error' && (
        <>
          <WarningIcon size={12} />
          Save failed
        </>
      )}
    </div>
  );
}
```

Key changes:
1. Remove `useState` / `useEffect` / delayed show logic entirely
2. Treat `idle` the same as `saved` — shows a muted green "Saved" indicator
3. `saving` shows amber immediately (no 400ms wait)
4. Content never disappears — bar always has visible text

**Done:** Status bar always shows text. Transitions cleanly between Saving (amber) and Saved (muted green). No flash or disappearing.
