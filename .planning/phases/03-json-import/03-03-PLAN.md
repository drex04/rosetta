2. In useCanvasData.ts: verify (or implement) that returned nodes array = `[...masterNodes, ...(activeSource?.schemaNodes ?? [])]` and edges = `[...masterEdges, ...(activeSource?.schemaEdges ?? [])]`.

3. Position persistence on source switch — two-filter approach (IMPORTANT: do not combine master and source change handling):
   - Build a `Set<string>` of source node IDs from `activeSource?.schemaNodes.map(n => n.id)`.
   - In `onNodesChange`: split incoming changes into two groups by node id membership in that Set.
   - Apply master-node changes (not in Set) via the existing ontologyStore handler (no change to that path).
   - Apply source-node changes (in Set) by calling `applyNodeChanges(sourceChanges, activeSource.schemaNodes)` then `updateSource(activeSourceId, { schemaNodes: updatedNodes })`.
   - If `activeSource` is null or the Set is empty, skip the source path entirely.

4. Ensure switching active source (setActiveSourceId) causes canvas to re-render with new source's schemaNodes at their stored positions — this should work automatically via useCanvasData if the store is reactive.

5. In useCanvasData.ts: update the type cast to use the new `SourceNode` type (imported from `@/types/index`). Remove the comment `// source nodes typed properly in a later phase`. Cast `active?.schemaNodes` as `SourceNode[]` and spread them correctly. Do NOT change the runtime behavior — type-only fix.
  </action>
