## Decision 1: Comunica lazy-loaded via dynamic import
**Category:** architecture
**What:** `fusion.ts` uses `await import('@comunica/query-sparql')` with a module-level singleton engine. Vite code-splits Comunica into a separate chunk.
**Why:** Comunica adds ~3–5 MB to the bundle. Deferring load until first "Transform & Fuse" click prevents blocking initial app paint.
**Alternatives considered:** Static import (unacceptable initial load); N3.Store pattern matcher only (cannot handle user-edited SPARQL).
**How to apply:** Any future import of Comunica in app code must also use dynamic import unless the build is split differently.

---

## Decision 2: fusionStore is ephemeral (no IDB persistence)
**Category:** architecture
**What:** Fusion results are lost on page reload. User re-runs fusion explicitly.
**Why:** Stale-invalidation logic for persisted RDF output is complex — mappings or source data can change at any time. Mirrors the `validationStore` precedent.
**Alternatives considered:** IDB-persist fused JSON (adds stale-invalidation complexity for no durable benefit).
**How to apply:** Do not add IDB persistence to `fusionStore` without addressing staleness.

---

## Decision 3: Provenance via `prov:wasAttributedTo` literal
**Category:** architecture
**What:** One `<entity> prov:wasAttributedTo "<sourceName>"^^xsd:string` triple added per distinct subject per source after fusion.
**Why:** Simple, self-contained, works with JSON-LD compaction. Full PROV-O model is over-engineered for a learning tool.
**Alternatives considered:** Named graphs (breaks JSON-LD serialization); full PROV-O (over-engineered).
**How to apply:** If named graph support is added later, provenance annotations will need to migrate.

---

## Decision 4: JSON-LD compaction, not full framing
**Category:** architecture
**What:** `jsonldFramer.ts` uses `jsonld.fromRDF()` + `jsonld.compact()` with auto-derived context. No `jsonld.frame()`.
**Why:** Full framing requires a user-defined frame document that is brittle with auto-derived types from arbitrary ontologies.
**Alternatives considered:** Full framing (deferred to a future "frame editor" feature — see Deferred Ideas in CONTEXT.md).
**How to apply:** Context is auto-derived from OntologyNode URIs; keep this simple.

---

## Decision 5: Mapping kind picker is inline, not modal
**Category:** product
**What:** Kind-specific fields (templatePattern, constantValue, targetDatatype, languageTag, join fields) render inline in MappingPanel below the kind `<select>`.
**Why:** Modal/drawer breaks the panel flow in narrow right-panel layout. Panel is already constrained — inline fields keep context visible.
**Alternatives considered:** Modal/drawer (rejected for layout reasons).
**How to apply:** Keep all mapping-related UI inline in the panel, not in overlays.

---

## Decision 6: `join` kind generates a placeholder SPARQL stub with `FILTER(false)`
**Category:** architecture
**What:** `generateConstruct` for `kind === 'join'` emits a commented stub with `FILTER(false)` guard so it is syntactically valid but returns no data.
**Why:** Heuristic join SPARQL generation has high risk of silent semantic errors. Better to make the limitation explicit.
**Alternatives considered:** Heuristic join SPARQL (rejected — silent errors worse than no-op).
**How to apply:** RML/YARRRML also annotate join mappings as `# requires manual conversion`.

---

## Decision 7: RML subject template uses `*id*`/`*key*` property heuristic
**Category:** implementation
**What:** First property whose local name contains `id` or `key` is used as template variable; falls back to first property; falls back to `http://example.org/{classLocalName}/{index}` if no properties.
**Why:** No blank node templates in RML — `rr:template` only generates IRIs.
**Alternatives considered:** Blank node (`_:r{trackId}`) — removed because `rr:template` is IRI-only.

---

## Decision 8: YARRRML is a plain string — no js-yaml dependency
**Category:** implementation
**What:** YAML built with manual string interpolation; double-quote values; escape `"` as `\"`.
**Why:** js-yaml adds ~60 KB to bundle. YARRRML structure is shallow and regular; manual interpolation is straightforward.
**Alternatives considered:** js-yaml (rejected — unwarranted bundle cost).

---

## Decision 9: OUT tab gets three sub-tabs: Ontology | Fused | Export
**Category:** product
**What:** Ontology retains existing Turtle/JSON-LD toggle. Fused shows fusionStore output + Transform button. Export shows RML/YARRRML downloads.
**Why:** Keeps all output in one panel tab. No RightPanel layout changes needed.
**Alternatives considered:** Separate top-level tabs (requires RightPanel layout restructuring).
**How to apply:** If more output types are added in Phase 7, extend the sub-tab list in OutputPanel.
